# Skill — Summarize Claim File

## Purpose

Take a long, messy claim file (notes, emails, call transcripts, documents) and produce a handler-ready brief. Useful when a claim is being picked up by a new handler, escalated, or reviewed.

## Input

A claim file, typically 5–50 pages of mixed content — chronological notes, inbound and outbound correspondence, photos (described), expert reports, internal comments.

## Output

```
ONE-LINE SUMMARY
(who, what, when, current status)

TIMELINE
(dated bullet list — only material events, not every note)

COVERAGE POSITION
(current preliminary view, clause references)

FINANCIALS
(claim amount notified, any payments made, outstanding — but NEVER include or estimate a reserve)

OUTSTANDING ACTIONS
(who owns what, deadlines)

RISK FLAGS
(fraud, vulnerability, complaint, litigation, media, large-loss, regulator)

WHAT I'VE LEFT OUT AND WHY
(one-line explanation of what a fuller review would add)
```

## Hard rules

- **No reserve information.** If the file contains reserve figures, do not surface them in the summary. They are for internal systems, not briefs.
- Quote exact figures the customer has notified, not estimates.
- If the file is inconsistent with itself, flag that explicitly — don't silently pick a version.
- Redact personal data beyond what's needed to identify the claim.
- Note your confidence in the summary. If the file has gaps, say so.

## Length target

One page. If the handler needs the full file, they'll open it. The summary's job is orientation, not replacement.
